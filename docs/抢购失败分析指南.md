## 🎯 第一次抢购失败后如何分析

### 第一步：找到日志文件

脚本运行后会自动生成日志文件：

```
logs/auto-buy_2026-04-09T10-00-05-123Z.log
```

文件名格式：`auto-buy_YYYY-MM-DDTHH-mm-ss-SSSZ.log`

**查看最新日志**：
```powershell
# Windows PowerShell
Get-Content logs\auto-buy_*.log -Tail 100
```

### 第二步：快速诊断（30 秒）

在日志文件中查找关键信息：

#### 1. 查找错误
```powershell
Select-String -Path logs\auto-buy_*.log -Pattern "\[ERROR\]" -Context 2
```

**看到 `[ERROR]` 表示**：
- 网络请求失败
- 元素找不到
- 点击失败
- 页面加载超时

#### 2. 查看库存状态变化
```powershell
Select-String -Path logs\auto-buy_*.log -Pattern "库存状态"
```

**期望看到**：
```
[10:00:00.123] [INFO] [STOCK] 库存状态：SOLD_OUT
[10:00:00.456] [INFO] [STOCK] 库存状态：AVAILABLE ← 库存释放！
```

**如果一直看到 `SOLD_OUT`**：
- 库存尚未释放
- 需要继续等待或重试

#### 3. 查看点击尝试
```powershell
Select-String -Path logs\auto-buy_*.log -Pattern "第.*次尝试"
```

**期望看到**：
```
[10:00:00.567] [INFO] [ATTEMPT] 第 1 次尝试...
[10:00:00.789] [SUCCESS] [CLICK] 点击 Pro 套餐按钮成功
```

**如果看到多次尝试**：
- 第一次点击失败
- 系统在自动重试

### 第三步：详细分析（5 分钟）

#### 场景 A：库存一直显示售罄

**日志特征**：
```
[10:00:00.123] [INFO] [STOCK] 库存状态：SOLD_OUT
[10:00:00.325] [INFO] [STOCK] 库存状态：SOLD_OUT
[10:00:00.527] [INFO] [STOCK] 库存状态：SOLD_OUT
...
[10:00:30.891] [ERROR] [RESULT] 抢购失败：达到最大重试次数
```

**可能原因**：
1. **库存尚未释放** - 服务器时间可能比你的时间慢
2. **瞬间售罄** - 库存释放后几毫秒内被抢完
3. **选择器错误** - 页面结构变化，找不到按钮

**解决方案**：
- 提前 1-2 分钟开始（修改 `BUY_HOUR: 9, BUY_MINUTE: 58`）
- 增加重试次数（`MAX_RETRIES: 50`）
- 运行测试脚本确认选择器正确

#### 场景 B：点击失败

**日志特征**：
```
[10:00:00.456] [INFO] [STOCK] 库存状态：AVAILABLE
[10:00:00.567] [INFO] [CLICK] 尝试点击 Pro 按钮...
[10:00:00.789] [ERROR] [CLICK] 点击失败：元素不可见
```

**可能原因**：
1. **按钮仍被遮罩层覆盖** - 页面未完全加载
2. **元素选择器错误** - ref 或 class 已变化
3. **页面状态未更新** - DOM 已变但视觉未变

**查看截图**：
```
screenshots/error_click_failed_20260409_100000.png
```

**解决方案**：
- 增加页面等待时间
- 运行测试脚本获取最新选择器
- 查看截图确认页面实际状态

#### 场景 C：网络请求失败

**日志特征**：
```
[10:00:00.456] [INFO] [NETWORK] POST https://www.bigmodel.cn/biz/pay/preview
[10:00:05.678] [ERROR] [NETWORK] 请求失败：timeout
```

**可能原因**：
1. **服务器过载** - 10:00 时大量请求涌入
2. **网络延迟** - 你的网络连接问题
3. **防火墙/代理** - 被拦截

**解决方案**：
- 使用更快的网络
- 增加超时时间（`PAGE_LOAD_TIMEOUT: 120000`）
- 检查防火墙设置

#### 场景 D：进入支付页面失败

**日志特征**：
```
[10:00:00.789] [SUCCESS] [CLICK] 点击成功
[10:00:01.234] [INFO] [PAYMENT] 检查支付页面...
[10:00:03.456] [WARNING] [PAYMENT] 未检测到支付页面
[10:00:03.567] [INFO] [ATTEMPT] 第 2 次尝试...
```

**可能原因**：
1. **点击了但没反应** - 按钮事件未触发
2. **跳转延迟** - 页面跳转慢
3. **弹窗拦截** - 浏览器拦截了弹窗

**查看截图**：
```
screenshots/after_click_20260409_100000.png
```

**解决方案**：
- 增加支付页面等待时间
- 禁用浏览器弹窗拦截
- 查看截图确认点击后状态

## 📊 日志类别说明

日志按类别组织，便于过滤：

| 类别 | 说明 | 关键信息 |
|------|------|----------|
| `STOCK` | 库存状态检查 | `SOLD_OUT` / `AVAILABLE` |
| `STOCK_WATCH` | 实时监控 | 每秒检查状态 |
| `CLICK` | 按钮点击 | 成功/失败，元素状态 |
| `ATTEMPT` | 抢购尝试 | 第几次尝试 |
| `PAYMENT` | 支付页面检测 | 是否成功进入 |
| `NETWORK` | 网络请求 | URL、状态码、耗时 |
| `CONSOLE` | 浏览器控制台 | JS 错误、警告 |
| `AUTH` | 登录状态 | 已登录/未登录 |
| `ERROR` | 错误详情 | 完整堆栈、上下文 |
| `SCREENSHOT` | 截图记录 | 文件路径 |

```powershell

Select-String -Path logs\auto-buy_*.log -Pattern "\[ERROR\]" -Context 3

# 查看库存状态变化

Get-Content logs\auto-buy_*.log | Select-Object -Last 100

# 实时查看日志（脚本运行时）
Get-Content logs\auto-buy_*.log -Tail 20 -Wait
```

```bash
# 查看所有错误
grep "\[ERROR\]" logs/auto-buy_*.log -A 3

# 查看时间线
tail -n 100 logs/auto-buy_*.log

# 实时查看
tail -f logs/auto-buy_*.log
```

## 🎯 典型失败案例分析

### 案例 1：库存瞬间售罄

**日志**：
```
[09:59:59.890] [INFO] [STOCK] 库存状态：SOLD_OUT
[10:00:00.012] [INFO] [STOCK] 库存状态：AVAILABLE  ← 库存释放！
[10:00:00.234] [INFO] [CLICK] 尝试点击...
[10:00:00.456] [WARNING] [STOCK] 库存状态：SOLD_OUT  ← 又被抢完了！
[10:00:00.678] [ERROR] [CLICK] 点击失败：按钮已禁用
```

**分析**：
- 库存在 10:00:00.012 释放
- 0.234 秒后尝试点击
- 但 0.456 秒时又售罄了
- **抢购太激烈，手速不够**

**解决方案**：
- 脚本已经是最快速度（200ms 重试）
- 可能需要增加并发（但可能被封锁）
- 尝试在非高峰期的其他时间捡漏

### 案例 2：页面未响应

**日志**：
```
[10:00:00.123] [INFO] [STOCK] 库存状态：SOLD_OUT
[10:00:00.345] [INFO] [STOCK] 库存状态：AVAILABLE
[10:00:00.456] [INFO] [CLICK] 尝试点击...
[10:00:05.678] [ERROR] [CLICK] 点击超时：5000ms
[10:00:05.890] [ERROR] [NETWORK] 无网络请求记录
```

**分析**：
- 库存已释放
- 点击按钮但无响应
- 页面可能卡死或 JS 错误

**查看浏览器控制台日志**：
```powershell
Select-String -Path logs\auto-buy_*.log -Pattern "\[CONSOLE\]" -Context 1
```

**可能看到**：
```
[CONSOLE] ERROR: Uncaught TypeError: Cannot read property 'click' of null
```

**解决方案**：
- 页面 JS 加载失败，需要刷新
- 脚本已配置定期刷新（每 10 次尝试）
- 检查网络连接

### 案例 3：登录状态丢失

**日志**：
```
[09:59:50.123] [INFO] [AUTH] 检测到已登录状态
[10:00:00.456] [INFO] [CLICK] 尝试点击...
[10:00:00.678] [WARNING] [AUTH] 检测到未登录状态
[10:00:00.890] [INFO] [AUTH] 等待用户手动登录...
```

**分析**：
- 开始时已登录
- 点击时 session 过期
- 可能被强制登出

**解决方案**：
- 检查账号是否在其他地方登录
- 确认 session 有效期
- 配置 `USER_DATA_DIR` 保持登录

## 🛠️ 调试工具

### 1. 日志查询 API

在脚本中添加：

```javascript
// 获取所有错误日志
const errors = logger.getLogs({ level: 'ERROR' });
console.log('错误数量:', errors.length);
errors.forEach(e => console.log(e.message));

// 获取网络请求日志
const networkLogs = logger.getLogs({ category: 'NETWORK' });
console.log('网络请求:', networkLogs.length);

// 获取状态变化
const stateChanges = logger.stateChanges;
console.log('状态变化:', stateChanges.length);
stateChanges.forEach(c => {
  console.log(`${c.element}: ${c.oldState} -> ${c.newState}`);
});
```

### 2. 导出日志为 JSON

```javascript
logger.exportLogs('json', './analysis.json');
```

然后用文本编辑器打开 `analysis.json` 进行详细分析。

### 3. 生成时间线

```javascript
const timeline = logger.getLogs();
timeline.forEach(log => {
  console.log(`${new Date(log.timestamp).toISOString()} [${log.level}] ${log.message}`);
});
```

## 🎯 优化建议

根据日志分析结果，可能的优化方向：

### 1. 时间优化
- **问题**：库存释放时脚本还没开始
- **解决**：提前 2 分钟启动（`BUY_HOUR: 9, BUY_MINUTE: 58`）

### 2. 速度优化
- **问题**：点击速度不够快
- **解决**：减少重试间隔（`RETRY_DELAY: 100`）

### 3. 稳定性优化
- **问题**：页面响应慢或卡死
- **解决**：增加定期刷新频率（每 5 次尝试刷新）

### 4. 选择器优化
- **问题**：找不到按钮或选择器错误
- **解决**：运行测试脚本更新选择器

### 5. 网络优化
- **问题**：网络请求超时
- **解决**：增加超时时间，使用更快网络
